# PRD：个人思考成长平台（Personal Growth OS）

**版本**: v1.0 Draft
**日期**: 2026-06-11
**目标读者**: 开发者本人（单人项目）

---

## 1. 问题陈述（Problem Statement）

自驱型学习者（如备战 ML/CV/Agent 方向求职、备考雅思）面临三个长期痛点：

1. **目标到执行的断层**：大目标（"拿到算法 offer"）无法自动转化为可执行的每日任务，拆解依赖人工且经常拆得不合理。
2. **学习闭环缺失**：学完一个主题后缺乏客观验证手段，"以为会了"和"真的会了"之间没有考核机制；薄弱点不会被系统性地回炉。
3. **信息收集成本高**：岗位要求、面经、考试技巧等外部信息需要每天手动搜集整理，重复劳动多。

不解决的代价是：学习计划流于形式、知识盲区持续累积、求职/考试准备效率低下。

**核心理念**：这是一个 **掌握式学习（Mastery Learning）系统**——任务不以"时间到了"为结束标志，而以"AI 考核通过"为结束标志。不达标则任务延期（delay），并基于错题增量重拆解子任务：**已掌握的部分保留，薄弱的部分加强**。

---

## 2. 目标（Goals）

| # | 目标 | 衡量方式 |
|---|------|---------|
| G1 | 大目标可被结构化为「阶段目标 → 阶段任务 → 子任务」三层，且子任务由 AI 自动拆解 | 创建一个阶段目标到生成可执行子任务列表 ≤ 5 分钟 |
| G2 | 每个阶段任务有客观出口：AI 生成考试 → AI 评估 → 通过才能关闭 | 100% 阶段任务必须经过考核才能标记完成 |
| G3 | 不达标任务自动 delay 并增量重拆解（保留已掌握、强化薄弱点） | 重拆解后已通过的子任务保留率 100%，新增子任务与错题知识点对应 |
| G4 | 外部信息收集自动化：cronjob 定时跑 AI 生成的脚本采集数据 | 配置一个采集任务（描述需求 → 生成脚本 → 测试 → 上线）≤ 10 分钟 |
| G5 | 学习全过程可记录、可回溯：笔记随时记、随时查，与目标/任务关联 | 任意笔记 3 秒内可通过目标/任务/全文检索找到 |

## 3. 非目标（Non-Goals）

| 非目标 | 原因 |
|--------|------|
| 多用户 / 团队协作 / 权限系统 | v1 为单人自用，多租户会让数据模型和部署复杂度翻倍 |
| 移动端 App | Web 响应式够用，学习场景以桌面为主 |
| 内置题库 / 内置学习内容 | 内容全部由 AI 按需生成或脚本采集，不做内容运营 |
| 自训练/微调模型 | 仅通过 API 接入外部 LLM（OpenAI 兼容协议） |
| 社交、分享、排行榜 | 与"个人思考成长"定位无关 |
| 复杂工作流编排（DAG、条件分支） | cronjob 是简单的「定时 → 跑脚本 → 存结果」，不做 Airflow |

## 4. 用户画像与核心场景

**用户**：开发者本人。具备 STEM 背景，会写代码，能读懂 AI 生成的脚本，对 AI 输出有判断力。

### 场景 A：求职准备（机器学习/CV/算法/AI Agent 方向）

1. 创建阶段目标「毕业后拿到 ML/CV/算法/Agent 相关 offer」，时间范围 6 个月，优先级 P0。
2. 配置定时任务「每日岗位情报」：每天 08:00 跑脚本采集目标岗位的 JD、技能要求、面经，AI 整理成当日学习材料和面试题清单，推送到仪表盘。
3. 创建阶段任务「掌握 DETR」，截止日期 2 周后。AI 将其拆解为子任务：Transformer 复习 → 二分图匹配与匈牙利算法 → DETR 架构精读 → 损失函数推导 → 代码复现。
4. 逐个完成子任务，过程中随手记笔记。
5. 截止日到达，触发考试：AI 按配置的 Prompt（如"按大厂算法面试难度出题"）生成测试题。
6. 作答后 AI 评估：匈牙利算法部分答错 → 任务不通过 → delay 一周 → 子任务重拆解：已通过的「Transformer 复习」「架构精读」标记为已掌握并保留，新增「匈牙利算法手推 + 实现」「匹配代价函数细节」两个强化子任务。
7. 补强后再次考试，通过 → 任务关闭。

### 场景 B：雅思备考（目标 7 分）

1. 创建阶段目标「雅思 7 分」，创建阶段任务「阅读练习」「作文练习」。
2. 子任务为不同题型的练习（如 T/F/NG 题、Matching 题、Task 2 议论文），做完对照解析学习。
3. 定时任务：脚本定期采集高频考点、技巧文章、真题。
4. 任务结束触发模拟测试，AI 评分。不达标打回：阅读中 Matching 题正确率低 → 保留其他题型子任务的已掌握状态 → 加练 Matching 专项。

## 5. 功能需求

### 5.1 信息架构（五大模块）

```
首页（仪表盘）
├── 阶段管理（目标 / 任务 / 子任务 / 考试 / 任务级 cronjob）
├── 笔记管理（全局笔记 + 目标关联笔记）
├── 脚本管理（所有定时脚本：任务级 + 独立）
└── 系统配置（AI 接入、Prompt 模板、全局设置）
```

### 5.2 首页 — 仪表盘

**P0（必须有）**

- **今日聚焦**：今天到期/进行中的子任务、待考试的阶段任务、今日 cronjob 产出（如当日采集的岗位情报摘要）。
- **目标进度总览**：每个进行中阶段目标的完成度（按子任务加权）、剩余天数、delay 次数。
- **数据统计**：
  - 学习活跃度：近 30 天每日完成子任务数 / 笔记数（热力图）。
  - 考试统计：通过率、平均分、delay 次数趋势。
  - 知识薄弱点：从历次考试错题聚合出的高频薄弱知识点 Top N。
- **cronjob 运行状态**：最近运行的脚本及其成功/失败状态。

**P1**

- 周报/月报：AI 自动总结本周学习内容、薄弱点变化、下周建议。
- 仪表盘卡片自定义排序与显隐。

**验收标准（节选）**

- [ ] 打开首页 2 秒内展示今日任务与目标进度
- [ ] 薄弱知识点可点击跳转到对应的错题与强化子任务
- [ ] cronjob 失败有醒目标识并可一键跳转到运行日志

### 5.3 阶段管理（核心模块）

#### 5.3.1 阶段目标（Goal）

**P0**

- 创建/编辑字段：**名称、描述、时间范围（开始/结束日期）、优先级（P0/P1/P2）、状态**（未开始/进行中/已完成/已归档）。
- 目标详情页聚合：其下所有阶段任务、关联 cronjob、关联笔记、进度统计。
- 删除采用归档（软删除），保留历史数据。

**验收标准**

- [ ] 时间范围校验：结束日期不得早于开始日期
- [ ] 目标进度 = 其下阶段任务进度的加权平均（权重可配置，默认等权）
- [ ] 归档目标不出现在仪表盘，但可在归档列表查看与恢复

#### 5.3.2 阶段任务（Stage Task）

**P0**

- 字段：**任务目标内容（学什么、达到什么程度）、时间范围、完成度（自动按子任务计算，可手动校准）、所属阶段目标、考试配置**。
- **考试配置**：出题 Prompt 模板（可选内置模板或自定义，如"按一线大厂算法面试难度出 8 道题，含 2 道代码题"）、题量、通过阈值（如 80 分）、最大 delay 次数。
- **生命周期状态机**（详见技术文档）：
  `待开始 → 进行中 → 待考试 → 考试中 → 评估中 → 已通过 / 未通过(delay)`
- **delay 机制**：考试不达标 → 任务自动延期（用户确认延期时长，AI 给出建议）→ 触发子任务重拆解 → 回到「进行中」。
- delay 次数与历史考试记录全程留痕。

**P1**

- 达到最大 delay 次数后提示用户：缩小任务范围 / 调低通过阈值 / 强制关闭（需手动确认并留痕）。
- 任务模板：常用任务结构（如"论文精读"模板）可保存复用。

**验收标准**

- [ ] 任务到达截止时间且子任务全部完成 → 自动进入「待考试」并通知用户
- [ ] 子任务未全部完成而到期 → 提示用户选择：直接考试 / 先 delay
- [ ] 「已通过」为终态，不可再编辑子任务（只读留档）
- [ ] 每次 delay 记录：考试得分、错题、AI 评语、延期时长

#### 5.3.3 子任务（SubTask）

**P0**

- **AI 拆解**：输入阶段任务的目标内容 + 时间范围，AI 输出子任务列表（标题、学习内容说明、预估耗时、建议顺序、关联知识点标签）。用户可对 AI 结果**增删改、调整顺序后确认**——AI 拆解是草稿，用户确认才生效。
- 手动标记完成（完成度由用户勾选驱动，系统不自动判定）。
- 子任务带**掌握状态**：`未开始 / 进行中 / 已完成 / 已掌握（考试验证）/ 薄弱（考试暴露）`。
- **增量重拆解**（考试不达标后）：
  - AI 输入：原子任务列表 + 各子任务掌握状态 + 错题及其知识点。
  - 规则：**已掌握的子任务保留且锁定**（不删除答对的部分）；薄弱知识点对应的子任务被标记为「薄弱」，AI 生成针对性的强化子任务（查缺补漏）；可新增此前遗漏的知识点子任务。
  - 输出为 **diff 形式**（保留 / 强化 / 新增），用户确认后应用。

**P1**

- 子任务支持二级嵌套（子任务下的 checklist）。
- 每个子任务可一键发起「针对该子任务的小测」（不影响任务状态，纯自测）。

**验收标准**

- [ ] AI 拆解结果用户可全量编辑后再确认，未确认不写入
- [ ] 重拆解后，考试中答对知识点对应的子任务状态为「已掌握」且不被删除
- [ ] 重拆解后，每个错题至少映射到一个强化子任务
- [ ] 子任务详情可查看其关联的历史笔记与历史考题

#### 5.3.4 考试与 AI 评估（Exam）

**P0**

- **触发**：阶段任务到期或用户手动发起。
- **生成**：按任务的考试配置（Prompt 模板 + 变量注入：任务目标、子任务列表、历史错题、用户笔记摘要）调用 LLM 生成题目。题型支持：单选、多选、简答、代码题。
- **作答**：网页内作答界面，支持 Markdown 与代码块；可中途保存草稿。
- **评估**：AI 按预生成的**评分标准（rubric，与题目同时生成）**逐题打分，输出：总分、每题得分、错因分析、薄弱知识点列表、是否通过。
- **人工 override**：用户可对任意一题的 AI 判分提出复核（附理由），手动改分并留痕——AI 评估主观题可能不稳定，必须有人工兜底。
- 历史考试记录：所有题目、作答、评分、评语永久保留，可回看。

**P1**

- 错题本：跨任务聚合所有错题，支持按知识点筛选、重做。
- 出题去重：同一任务多次考试时，避免与历史题目重复（将历史题摘要注入 Prompt）。

**验收标准**

- [ ] 题目与评分 rubric 同时生成并持久化（评估时使用生成时的 rubric，保证判分一致性）
- [ ] 总分 ≥ 通过阈值 → 任务进入「已通过」；否则进入 delay 流程
- [ ] AI 评估失败（API 错误/格式不合法）→ 考试停留在「评估中」可重试，不丢失作答
- [ ] 人工改分后重新计算总分与通过状态，并标注「含人工复核」

#### 5.3.5 任务级定时任务（cronjob）

**P0**

- 在阶段目标/任务下配置 cronjob：**触发时间（cron 表达式，提供可视化配置器 + 常用预设）、需求描述、AI 生成的脚本、启用开关**。
- **AI 生成脚本流程**：用自然语言填写需求（如"每天采集 LinkedIn/BOSS 直聘上 CV 算法岗的 JD 和要求，汇总成 Markdown"）→ AI 生成脚本（Python）→ 用户审阅代码 → **点击「测试运行」**查看输出 → 确认后保存启用。
- 脚本在**沙箱中执行**（资源、时长、网络受限，详见技术文档），运行结果（stdout / 产物文件 / 错误日志）持久化并在任务详情和仪表盘展示。
- 运行历史：每次触发的时间、时长、状态、输出快照。

**P1**

- 脚本输出可配置「后处理 Prompt」：原始采集结果再过一遍 LLM 整理（如"从这些 JD 中提取技能要求频次 Top 20 并生成今日学习建议"）。
- 失败重试策略（最多 N 次，间隔退避）与失败通知。

**验收标准**

- [ ] cron 表达式有合法性校验与「下次运行时间」预览
- [ ] 测试运行与正式运行走同一沙箱路径，结果一致
- [ ] 脚本修改后旧版本留存（版本历史），可回滚
- [ ] 单次运行超时（默认 5 分钟，可配）自动终止并标记失败

### 5.4 笔记管理

**P0**

- **两种来源**：① 学习过程中在目标/任务/子任务/考试页面内随手创建（自动关联当前上下文）；② 笔记模块内独立创建（不关联任何目标）。
- Markdown 编辑器（代码高亮、图片粘贴、LaTeX 公式——学习场景刚需）。
- 列表与检索：按关联目标/任务筛选、标签筛选、全文搜索、时间排序。
- 历史笔记在对应任务/子任务详情页内嵌展示（"查看历史笔记"）。

**P1**

- 笔记版本历史。
- AI 辅助：选中笔记一键「总结」「生成自测题」。
- 双链引用（笔记间互相链接）。

**验收标准**

- [ ] 在子任务页创建的笔记自动带上目标/任务/子任务三级关联，且可后期修改关联
- [ ] 全文搜索覆盖标题与正文，1 万条笔记内响应 < 1 秒
- [ ] 删除为软删除，回收站可恢复（30 天）

### 5.5 脚本管理

**P0**

- **统一视图**：聚合全部定时脚本——来自阶段管理的任务级 cronjob + 独立创建的全局脚本（如"每周备份数据库""每天抓 arXiv 新论文"）。
- 独立脚本创建流程与任务级一致：需求描述 → AI 生成 → 审阅 → 测试运行 → 启用。
- 列表展示：名称、来源（关联目标 or 独立）、cron 表达式、上次运行状态、下次运行时间、启停开关。
- 脚本编辑器（代码高亮）+ 运行日志查看。

**P1**

- 脚本依赖声明（requirements）与沙箱环境预装管理。
- 手动脚本（无 cron，仅手动触发的工具脚本）。

**验收标准**

- [ ] 任务级与独立脚本在此模块统一可见、可启停、可测试
- [ ] 删除关联了任务的脚本时给出警告并要求确认

### 5.6 系统配置

**P0**

- **AI 接入配置**：API Host（Base URL）、API Key、模型名——兼容 OpenAI 协议，可接 OpenAI / DeepSeek / 本地推理服务（如 Mac mini 上的本地模型）等。支持配置多个 Provider 并指定默认。
- **分场景模型路由**：拆解 / 出题 / 评估 / 脚本生成 / 笔记辅助 五个场景可分别指定 Provider+模型（如评估用强模型、整理用便宜模型）。
- **连通性测试**：保存前一键测试（发送最小请求验证 key/host/model 可用）。
- API Key 加密存储，前端永远只显示掩码。
- **Prompt 模板管理**：内置模板（任务拆解、出题、评估、重拆解、脚本生成）可查看、复制、自定义覆盖；模板支持变量插值（`{{task_goal}}`、`{{subtasks}}`、`{{wrong_questions}}` 等），变量说明文档内置。

**P1**

- 用量统计（按场景的 token 消耗）。
- 数据导出/导入（全量 JSON 备份）。

**验收标准**

- [ ] 未配置 AI 时，所有 AI 功能入口给出清晰引导而非报错
- [ ] 连通性测试失败展示具体错误（401/超时/模型不存在）
- [ ] 自定义 Prompt 模板缺失必需变量时保存前校验并提示

## 6. 关键产品决策与理由

1. **AI 输出全部"草稿 + 人工确认"**：拆解、重拆解、脚本生成均不直接生效，用户确认后才写入。理由：LLM 输出质量不稳定，单人系统中用户即专家，确认成本低、纠错价值高。
2. **考试通过是任务唯一正常出口**：宁可 delay 也不允许"到期自动完成"，这是系统区别于普通 Todo 工具的核心；但提供最大 delay 次数 + 强制关闭的逃生通道，避免死循环挫败感。
3. **重拆解是 diff 不是覆盖**：保留已掌握内容既是需求明确要求，也避免学习记录断裂——掌握状态是系统最有价值的长期数据。
4. **脚本必须沙箱执行**：即使单人自用，AI 生成的代码也不可信任（可能误删文件、死循环、泄露数据），沙箱是 P0 不是 P1。
5. **rubric 与题目同时生成并固化**：避免评估时 LLM 重新"理解"题目导致判分漂移。

## 7. 成功指标

| 类型 | 指标 | 目标 |
|------|------|------|
| 先行 | 周活跃天数（有完成子任务或笔记的天数） | ≥ 5 天/周 |
| 先行 | 阶段任务考试首次通过率 | 40%–70%（过高说明题太简单，过低说明拆解不合理） |
| 先行 | cronjob 成功率 | ≥ 95% |
| 滞后 | 阶段目标按期（含合理 delay）完成率 | ≥ 70% |
| 滞后 | 薄弱知识点二次考核通过率 | ≥ 80%（验证"回炉"机制有效） |

## 8. 开放问题

| 问题 | 类型 | 影响 |
|------|------|------|
| AI 评估代码题：仅静态判读，还是沙箱实际运行测试用例？ | 工程 | v1 建议静态判读 + rubric，运行测试用例放 P2 |
| 考试作答是否限时？ | 产品 | v1 不限时（开卷自测定位），但记录作答时长 |
| 脚本需要访问需登录的网站（如 BOSS 直聘）怎么办？ | 工程/合规 | v1 仅支持公开数据源；登录态采集涉及反爬与账号风险，明确不做 |
| delay 时长由 AI 建议还是固定规则？ | 产品 | AI 建议（基于错题量），用户最终确认 |

## 9. 里程碑建议（三期交付）

| 阶段 | 范围 | 价值验证点 |
|------|------|-----------|
| **M1（核心闭环）** | 目标/任务/子任务 CRUD + AI 拆解 + 考试生成与评估 + delay 重拆解 + AI 配置 | 跑通一个完整的「学习 → 考核 → 回炉」循环 |
| **M2（自动化）** | cronjob 调度 + AI 生成脚本 + 沙箱执行 + 脚本管理 + 仪表盘 v1 | 每日岗位情报自动化场景可用 |
| **M3（沉淀）** | 笔记管理完整版 + 错题本 + 薄弱点统计 + 周报 | 长期数据开始产生复利 |

> 强烈建议严格按此顺序：M1 的掌握式闭环是系统存在的理由，cronjob 和仪表盘没有它就只是普通爬虫面板。
